# 992. Subarrays with K Different Integers

## 我的思考
首先识别题型。  
这是子数组计数问题，可以考虑滑动窗口。

初始思路是使用一个变长滑动窗口遍历数组，`map` 记录每个数字出现的次数。窗口的含义是：**窗口内部保持合法**。

但很快发现，这题和普通“求是否存在”不同，它要求的是**以每个 `right` 结尾的合法子数组个数**。例如：

- `nums = [1,2,1,3]`
- `k = 3`

当 `left = 0, right = 3` 时，实际上有不止一个合法答案。  
这说明仅维护一个合法窗口还不够，还需要进一步记录“左侧冗余前缀”的范围。

最初的表述是：  
需要记录窗口内最靠左的、恰好使窗口合法的位置，记为 `i`。这样每次可以累加：

`ans += i - left + 1`

因为从 `left` 到 `i` 的任意起点，都能形成一个以 `right` 结尾、恰好包含 `k` 个不同整数的子数组。

于是整体逻辑变成：

- 对每个 `right`，维护一个窗口，使其恰好包含 `k` 个不同整数
- 同时把窗口左侧所有冗余重复元素尽量压缩掉
- 设压缩后第一个不能再删的位置为 `i`，那么从左边界到这个位置之间的所有起点都有效，本轮贡献就是对应的个数

在实现过程中，逐渐把这个思路具体化成：
- `left` 和 `right` 维护当前窗口
- `diffNums` 表示窗口内不同整数个数
- `redundantPre` 表示左侧冗余前缀的起点

也就是：  
`left` 与 `right` 维护恰好满足要求的最短子数组窗口，`redundantPre` 维护冗余前缀。

当 `right` 扩展进一个新元素时：
- 如果该数之前没出现过，则 `diffNums++`
- 然后放入 `counts`

之后分情况处理。

如果 `diffNums > k`，说明窗口中不同整数过多。  
此时由于是新引入了一个不同整数，所以只需要让 `left` 右移，移出一个不同整数，随后 `diffNums--`，并把 `redundantPre` 重置到新的 `left`。

然后，无论这次是否出现了新的不同整数，都要继续做一件事：**压缩左侧冗余重复元素**。  
即只要 `counts.get(nums[left]) > 1`，就不断右移 `left`，把这些重复元素去掉。

压缩完成后：
- 如果 `diffNums == k`
- 那么说明当前窗口已经是“恰好包含 `k` 个不同整数的最短合法窗口”
- 而 `redundantPre` 到 `left` 之间的所有起点都能形成合法子数组

所以当前轮的贡献就是：

`ans += left - redundantPre + 1`

一开始写出的版本里，这部分“压缩冗余”逻辑在两个分支里重复出现了。  
之后又做了一次优化，把流程整理为：

1. `right` 扩展
2. 如果 `diffNums > k`，先移出一个不同整数，并更新 `redundantPre`
3. 统一压缩冗余前缀
4. 如果 `diffNums == k`，统计答案

这样结构更清晰，可读性更好，但核心思路没有变化，时空复杂度也保持最优。

## 卡点
1. 需要额外维护“左侧冗余前缀”的信息，否则无法正确统计每轮贡献。
2. 初版实现中，“压缩冗余”逻辑出现了重复，说明流程划分还不够清晰。

## 关键点
1. 窗口内维护的是“恰好包含 `k` 个不同整数”的最短合法区间。
2. `diffNums` 用来记录当前窗口中不同整数的数量。
3. `redundantPre` 用来记录左侧冗余前缀，从而统计当前 `right` 下有多少个合法起点。
4. 左侧重复元素要尽量压缩，因为它们不会影响不同整数个数，但会影响答案个数。
5. 当 `diffNums == k` 时，本轮贡献为 `left - redundantPre + 1`。

## 复杂度
- 时间：`O(n)`
- 空间：`O(k)` 到 `O(n)`，取决于窗口内不同元素数量，使用 `HashMap` 记录频次

## 注意
1. 这题的难点不是“维护合法窗口”，而是“统计同一个右端点对应的多个合法左端点”。
2. `redundantPre` 的语义必须想清楚，否则最后的计数公式很容易写错。
3. 优化后的版本主要是**流程重构**，不是复杂度优化，但可读性明显更好。
