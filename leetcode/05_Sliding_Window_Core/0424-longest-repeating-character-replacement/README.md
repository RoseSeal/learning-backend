# 424. Longest Repeating Character Replacement

## 我的思考
首先识别出这是一个**变长滑动窗口**问题。

核心观察是：如果一个窗口内，数量最多的那个字母出现了 `maxCount` 次，那么其余字母如果都修改成这个字母，所需修改次数就是：

`窗口长度 - maxCount`

因此，不变量最初被表述为：

**窗口长度 - 窗口内数量最多的字母数 `<= k`**

也就是说，窗口内除了出现次数最多的字母之外，其余字母都可以在 `k` 次修改内被改成同一个字母。

接着会遇到一个实现问题。  
新的字母进入窗口后，这个字母的出现次数可能会成为新的最大值，所以需要维护窗口内各个字母的频率，并且能够知道当前出现次数最多的字母是谁。

使用 `int[] counts` 维护 26 个大写字母的出现次数，并额外记录一个 `maxCount`。  
每次 `right` 纳入新字母后：
1. 更新该字母在 `counts` 中的频次
2. 判断它是否成为新的最多字母数量，并更新 `maxCount`
3. 再通过不变量判断窗口是否需要收缩

但这里会出现一个关键问题：

当 `left` 右移、某个字母退出窗口后，`maxCount` 可能会减少。  
最直接的做法是每次减少后重新遍历 `counts` 数组，重新计算当前窗口的最大频次。虽然这里只是 `O(26)`，常数不大，但不是最优。

AI 的提示是：**`maxCount` 的减少需要记录吗？**

这直接终结了前面的思考。

关键突破在于：**`maxCount` 没必要严格表示“当前窗口内的最大频次”，它可以表示“历史上这个窗口扩张过程中出现过的最大频次”。**

这样一来，`right` 扩张时仍然正常更新 `maxCount`，但 `left` 收缩时，即使某个字符离开窗口导致真实最大频次下降，也**不回退** `maxCount`。

此时，`maxCount` 的语义已经变了：
- 不是“当前窗口内真实的最大字母数量”
- 而是“历史上出现过的最大字母数量”

相应地，不变量的语义也发生了变化。  
它不再表示：

**所有被保留下来的窗口都严格满足当前窗口内所需修改次数 `<= k`**

而变成：

**所有被保留下来的窗口，至少仍然有潜力成为全局最优答案**

如果某个窗口连这种潜力都没有，即 `right - left + 1 - maxCount > k`，那就直接收缩。  
而如果某个窗口因为 `maxCount` 偏大，导致它在数学上看起来“放宽了条件”，也没有关系，因为这种宽松只会让窗口暂时多留一会儿，不会导致最终答案错误。

本质上，`left` 的收缩不会错过真正有潜力成为全局最优的窗口。  
没有成为全局最优潜力的窗口，无需在意。

## 卡点
1. 一开始虽然抓到了正确的不变量，但默认把 `maxCount` 理解为“当前窗口内的真实最大频次”，于是卡在左指针收缩后如何维护它。
2. 第一反应是每次收缩后重新遍历 `counts` 求一次最大值，虽然是常数复杂度，但说明思路还没有完全到位。
3. 真正的卡点不是“如何精确维护当前最大值”，而是“这个最大值是否真的必须精确”。

## 关键点
1. 这是典型的**变长滑动窗口**问题。
2. 判定窗口是否合法的核心式子是：

   `窗口长度 - maxCount > k`

   若成立，说明窗口内需要修改的字符数超过了 `k`，窗口必须收缩。
3. `counts[26]` 用来维护字母频次。
4. `maxCount` 只在 `right` 扩张时更新，不在 `left` 收缩时回退。
5. `maxCount` 的语义不是“当前窗口真实最大频次”，而是“历史最大频次”。
6. 这种写法依然正确，因为即使 `maxCount` 偏大，也只会让某些无效窗口晚一点被收缩，不会漏掉真正的最优答案。

## 复杂度
- 时间：`O(n)`
- 空间：`O(1)`

## 注意
1. 这题最关键的不是滑动窗口本身，而是**接受 `maxCount` 可以不精确回退**。
2. 若把 `maxCount` 强行维护为“当前窗口真实最大值”，也能做，但思路更笨重。
3. 这道题里真正维持的不是“当前窗口一定严格合法”，而是“窗口仍然保留成为最优答案的潜力”。
4. `ans` 可以在每轮窗口收缩完成后更新，因为此时窗口已经是当前规则下保留下来的最大合法候选窗口。
