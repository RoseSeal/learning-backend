# 18. 4Sum

## 我的思考
初始思路是建立 `hash` 表。  
通过循环遍历前 3 个数，再查表确定第 4 个数，同时还要保证不是同一个数字。

但问题在于：**去重逻辑比较困难**。

之后想到一种优化思路：  
使用 `map` 记录两数之和，键为和，值为对应的两个数，即 `List<int[2]>`。然后再嵌套 `for` 去枚举另外两个数。为了去重，两个枚举过程都需要先排序，并配合类似：

`if(i > 0 && nums[i] == nums[i - 1]) continue;`

但这个方案的问题也比较明显：
1. 整体偏重，不容易快速写到 bug-free。
2. 去重逻辑仍然比较复杂。
3. 时间复杂度是 `O(n^2 * k)`，其中 `k` 是 `map` 中某个和对应列表的长度。最坏情况下，`k` 可以达到 `n^2`，虽然这种情况基本不太可能出现，但说明这个方案并不优。

最终思路还是回到更经典的做法：  
**先排序，嵌套两层 `for` 确定前两个数，再用双指针确定后两个数。**

也就是：
- 先预排序
- 第一层循环固定第 1 个数
- 第二层循环固定第 2 个数
- 剩余区间用双指针找第 3、第 4 个数
- 同时在三个层面处理去重

这样整体结构更清晰，也更容易控制正确性。

之后 AI 质疑了两个点：

1. `curr` 可能溢出
2. 剪枝错误

对此的判断是：

第一，`curr = nums[begin] + nums[end]` 不会溢出。  
题设已经写明：`-10^9 <= nums[i] <= 10^9`，因此两个该范围内的 `int` 相加，结果仍然在 `int` 范围内，不需要额外用 `long`。

第二，剪枝没有问题。  
数组已经排序，`nums[i]` 是当前四个数中最小的那个。既然 `(long) nums[i] * 4` 都已经大于 `target`，那么后面不可能再找到满足条件的四元组，因此这里直接 `break` 是正确的。

## 卡点
1. 初始的哈希思路能想到，但去重很难处理干净。
2. 两数和 `map` 的方案理论上可做，但实现偏重，且不容易快速写到稳定无 bug。
3. 需要明确双指针方案中的去重和剪枝是否正确。

## 关键点
1. 先排序，把问题转化为“固定前两个数 + 双指针找后两个数”。
2. 排序后可以自然处理去重：
   - `i` 去重
   - `j` 去重
   - `begin / end` 命中答案后继续跳过重复值
3. `restTarget = target - nums[i] - nums[j]` 用 `long`，避免减法过程中溢出。
4. `(long) nums[i] * 4 > target` 的剪枝成立，因为排序后 `nums[i]` 是当前可选四元组中的最小值。

## 复杂度
- 时间：`O(n^3)`
- 空间：`O(1)`，不计答案输出

## 注意
1. `curr = nums[begin] + nums[end]` 不会溢出，因为题目给定的单个元素范围保证了两个数之和仍在 `int` 范围内。
2. `restTarget` 需要用 `long`，因为 `target - nums[i] - nums[j]` 这一步可能溢出。
3. `if((long) nums[i] * 4 > target) break;` 这个剪枝没有问题，只是它不是最强剪枝，但一定不会漏解。
4. 这题的标准解法就是排序 + 双层枚举 + 双指针，重点不在想出花哨做法，而在于把去重、边界、剪枝一次写对。
