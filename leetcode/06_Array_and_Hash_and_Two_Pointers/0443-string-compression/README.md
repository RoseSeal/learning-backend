# 443. String Compression

## 我的思考
这题思路非常直接，就是顺序扫描。

维护当前压缩段的字符和出现次数：
- 如果当前字符等于该段字符，就继续计数
- 否则就把这一段的字符和计数写回答案，并开启下一段

因此本质上就是“按段处理”的模拟题，没有什么额外隐藏坑。

一开始的写法是：
- 预先把段头保存下来
- 循环中处理当前段
- 同时准备下一段段头

这样会自然遇到一个问题：**最后一段结束后，循环退出了，但这一段还没被写入**，所以末尾需要额外补一个 `freq > 1` 的收尾判断。

之后想到能不能去掉这个末尾判断。最先想到的是加哨兵，但这会引入额外空间，不合适。

更自然的写法其实是另一种遍历方式：  
使用 `while` 按“段”来处理。

具体来说：
1. 先确定当前段的段头字符
2. 用内层 `while` 找到整段结尾，并统计频次
3. 这一段统计完成后，立刻把本段答案写入
4. 然后进入下一段

这样每一段都会在统计结束后立即完成写回，不再需要专门处理最后一段。

总结下来，这题虽然简单，但积累一个经验：

对于这种“**一段一段处理，并且在段尾统一收集结果**”的题，更自然的写法通常是：
- `while` 起段头
- `while` 找段尾
- 立即写入本段答案

而不是预先维护“当前段头 + 下一段段头”的切换逻辑。

## 关键点
1. 这是典型的按段扫描问题。
2. 每段只需要记录字符和出现次数。
3. 统计完一整段后，再统一写入答案更自然。
4. `write/read` 双指针写法比“预存段头 + 末尾补收尾”更顺。

## 复杂度
- 时间：`O(n)`
- 空间：`O(1)`

## 注意
1. 第一种写法的问题不在于错，而在于遍历方式不自然，所以末尾需要额外收尾。
2. 这类“按段处理”的题，优先考虑 `while` 找完整段，再一次性收集结果。
3. `Integer.toString(freq)` 这一步虽然生成了临时对象，但不影响题目的空间复杂度结论。
