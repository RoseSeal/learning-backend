# 127. Word Ladder

## 我的思考
一开始想到先建图再遍历。  
建图阶段，最初想直接根据 pattern 对单词分类，但这样不方便后续遍历，于是进一步想到：先按 pattern 分类，再根据分类结果构建邻接表。

遍历阶段，最初想到的是 DFS，因为脑中第一反应是判断是否存在路径。写完后才重新注意到题目要求的是最短路径，那么 DFS 会变得很绕，不适合实现，因此切换为 BFS。

具体做法是：  
先用 pattern 作为中间状态。例如把某一位替换成 `*`，将具有相同 pattern 的单词归到一组。然后基于这些分组，为每个单词建立邻接表。之后从 `beginWord` 出发做 BFS，层序扩展，当第一次遇到 `endWord` 时，当前层数就是答案。

## 卡点
1. 一开始遍历方式选成了 DFS，写到一半才意识到题目要求最短路径，应该直接用 BFS。  
2. 调试时出现了一个非常隐蔽的 bug：  
   原本写的是  
   `String pattern = s.substring(0, i) + s.substring(i + 1);`  
   少了通配符 `"*"`，导致 pattern 丢失了“变化发生在哪一位”的位置信息。  
   这样会把本不应该相连的单词判定为相连，例如 `tail` 和 `talk` 都可能被错误归到 `tal` 这一类中。  
3. 这题也暴露出对 String 操作仍然不够熟悉，涉及 `substring`、字符串拼接等操作时不够自然，导致实现和调试耗时较长。
4. bfs 时 visited 忘记标记。
5. 忘记把`beginWord`加入建图。

## 关键点
1. 这题本质是无权图最短路径问题，应该优先想到 BFS，而不是 DFS。  
2. pattern 的设计必须保留“变化发生的位置”这一信息，所以必须写成：
   `s.substring(0, i) + "*" + s.substring(i + 1)`  
   不能只是把某一位删除。  
3. 先通过 pattern 建立中间分组，再由分组结果构建邻接表，是一种可行做法。  
4. BFS 中第一次到达 `endWord` 时，对应的层数就是最短转换序列长度。

## 复杂度
- 时间：建 pattern 分组与建图都与单词数量和单词长度有关，之后 BFS 遍历图，整体优于直接两两比较建图。
- 空间：需要存储 pattern 分组、邻接表、visited 集合和 BFS 队列，空间复杂度与图的规模相关。
