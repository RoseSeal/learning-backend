# 417. Pacific Atlantic Water Flow

## 我的思考
一开始想到从答案格子本身出发思考。作为答案的格子有两种可能：一是能够直接流入 Pacific 和 Atlantic，二是能够流入已经判定为答案的格子。右上角格子 `[0][n - 1]` 和左下角格子 `[m - 1][0]` 可以视为初始答案。

随后考虑用深度搜索处理，并通过数组记录可达性，例如用 `1` 表示能流入 Pacific，`-1` 表示能流入 Atlantic，`0` 自然表示两边都能流入。但这种建模方式不够自然，实现时也会比较混乱。

意识到，二次搜索更好做。与其从每个格子正向判断它能否流到两边，不如分别从两侧海洋边界反向搜索，判断哪些格子可以“被 Pacific 反向到达”、哪些格子可以“被 Atlantic 反向到达”。最终同时被两次搜索到达的格子，就是答案。

实现上分别维护两个布尔数组：
- `toPO[i][j]` 表示该格子能够流入 Pacific；
- `toAO[i][j]` 表示该格子能够流入 Atlantic。

先将 Pacific 边界全部入队并做一次 BFS，再将 Atlantic 边界全部入队并做一次 BFS。最后遍历全图，取两个可达数组的交集即可。

## 卡点
一开始思路虽然较快收敛到“双边界反向搜索”，但在实现时没有先构思整体框架，而是直接开始写，导致先手写了两遍 BFS `while` 循环，后面才意识到结构重复，重新抽出了 `bfs` 方法。

此外又出现了低级错误：初始化 Atlantic 边界时，复制了 Pacific 的初始化循环，只把 `queuePO` 改成了 `queueAO`，却忘记将 `toPO[i][j] = true` 改成 `toAO[i][j] = true`，导致测试结果非常奇怪，额外花了约 10 分钟调试才找到问题。

## 关键点
- 正向判断“某格子是否能流到海洋”不自然时，可以考虑从目标区域反向搜索。
- 本题的核心不是从每个格子往海洋流，而是分别从 Pacific 和 Atlantic 边界出发，找出哪些格子能够反向到达这两个海洋。
- 思路一旦确定，最好先想清楚：
  - 需要几个状态数组；
  - 需要做几次 BFS/DFS；
  - 哪些逻辑可以抽成公共方法。
  这样可以减少重复代码和复制后漏改变量的低级错误。

## 复杂度
- 时间：`O(mn)`。Pacific 反向 BFS 一次 `O(mn)`，Atlantic 反向 BFS 一次 `O(mn)`，最后遍历交集一次 `O(mn)`，总复杂度仍为 `O(mn)`。
- 空间：`O(mn)`。两个布尔数组和 BFS 队列最坏情况下都可能达到 `O(mn)` 级别。

## 注意
- Pacific 边界是第一行和第一列，Atlantic 边界是最后一行和最后一列，初始化时要注意避免重复入队时的边界写法。
- 复制初始化代码后，要重点检查：
  - 写入的是不是正确的状态数组；
  - 使用的是不是正确的队列；
  - 边界行列是否对应正确。
- 这题 BFS 和 DFS 都可以做，关键不在遍历手法本身，而在“从海洋反向搜索”的建模转换。
