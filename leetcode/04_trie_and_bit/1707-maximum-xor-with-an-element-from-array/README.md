# 1707. Maximum XOR With an Element From Array

## 我的思考
对这题的理解是：每一个 `query = [x, m]`，本质上都是在 `nums` 中限定出一部分数字，然后在这一部分中找到与 `x` 的最大异或值。

暴力版本很直接：对每个 `query` 枚举所有满足 `<= m` 的数字，时间复杂度是 `O(M * N)`，其中 `M` 和 `N` 分别是 `queries` 和 `nums` 的长度。

接着思考如何优化。  
直觉上，`m` 这个约束本身不可能被消掉，因为必须知道哪些数字能参与查询。问题在于，`nums` 的遍历范围是否可以缩小。

于是想到：也许可以把 `query` 按 `m` 排序。这样随着 `m` 逐渐增大，可参与查询的 `nums` 前缀也会逐步扩大，而不必对每个 `query` 都重新扫描整个 `nums`。但这要求 `nums` 也有序。  
预处理的时间复杂度就会变成 `M log M + N log N`，当时第一反应是感觉不划算。

AI 提示：`O(N log N + M log M)` 比 `O(M * N)` 好得多，是划算的。

这样思路就能落地了。  
排序之后，随着 `query.m` 的增大，把所有 `<= m` 的 `nums` 依次加入 Trie（前缀树），每个查询只在当前 Trie 中寻找与 `x` 异或最大的数即可。显然这是 Trie 优化异或问题的标准模型。

需要注意的点有几个：

第一，`query` 排序之后原顺序会被打乱，因此还得保留原下标。  
一个自然的做法是新建 `query` 的副本，结构为 `int[queries.length][3]`，额外一位存原下标。

第二，Trie 查询时必须从最高位开始扫，因为高位优先级最高。  
例如当第 4 位和第 0 位发生冲突时，优先让第 4 位变成 `1`，因为它对最终异或值贡献更大。  
因此遍历时只要本位能走相反节点，就优先走相反节点。

第三，Trie 的位数可以有两种取法：
- 一种是直接固定取 `31`；
- 另一种是扫描最大值，找到其最高有效位，只遍历到那个 `bitPos`。

更倾向于后者。因为如果最大值只有几十、几百，直接扫到 31 位会带来一些时间和空间浪费。虽然不影响正确性，但实现上会显得不够紧。

第四，这题实现上有两种风格：
- 主流写法是 `bitPos` 直接取 `31`，或者取 `max(max(nums), max(queries[i][0]))`，然后在 Trie 的 `query` 函数里直接计算最大异或值；
- 另一种是这里采用的写法，`bitPos` 取 `max(nums)` 的最高位，`query` 函数只负责找出匹配到的那个数 `matchedNum`，最后再由 `queryValue ^ matchedNum` 计算异或结果。

## 卡点
1. 一开始虽然想到了“每个查询只看 `<= m` 的部分”，但没有立刻意识到可以通过按 `m` 排序，把这些部分组织成一个单调扩张的前缀。
2. 对排序预处理的代价有过犹豫，直觉上觉得 `M log M + N log N` 不一定划算，后来才明确它远优于暴力的 `O(M * N)`。

## 关键点
1. 将 `nums` 排序，将 `queries` 按 `m` 排序。
2. 按 `m` 从小到大处理查询，并用一个指针把所有 `<= m` 的 `nums` 依次加入 Trie。
3. 这样每个 `nums` 只会加入 Trie 一次，避免了重复筛选。
4. Trie 查询最大异或值时必须从高位到低位贪心，优先尝试走当前位相反的分支。
5. 由于查询被排序，必须保存原始下标，最后把答案填回对应位置。
6. 若当前没有任何 `nums[i] <= m` 被加入 Trie，则该查询答案为 `-1`。

## 复杂度
- 时间：`O(N log N + M log M + (N + M) * B)`，其中 `B` 是二进制位数；若视 `B` 为常数，则为 `O(N log N + M log M)`
- 空间：`O(M + N * B)`（若只计 Trie 额外空间，则为 `O(N * B)`）

## 注意
1. 这份实现里 `Trie` 是用 `nums` 最大值来确定 `bitPos` 的，因此默认只遍历到 `nums` 的最高有效位。
2. `query(int num)` 返回的不是“最大异或值”，而是 Trie 中与 `num` 最匹配的那个数本身，最终答案通过  
   `queryValue ^ trie.query(queryValue)`  
   得到。
3. 代码中 `if (numsIndex == 0)` 的含义是：当前还没有任何 `nums[i] <= m` 被加入 Trie，此时答案应为 `-1`。
4. 若采用固定 `31` 位的写法，实现会更统一，也更接近主流题解；当前写法则更偏向按实际数据范围裁剪位数，最求最高效率。
5. 这题的核心不是单个 Trie 查询，而是“排序 + 离线处理（offline query） + Trie”三者结合。
