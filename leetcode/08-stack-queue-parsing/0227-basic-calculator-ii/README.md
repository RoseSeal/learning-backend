# 227. Basic Calculator II

## 我的思考
一开始想到的可行思路太多，导致没有迅速收敛。

思路 1：使用栈。通用表达式求值可以用数字栈和运算符栈，但本题没有括号，因此双栈显得偏重，可以简化。
思路 2：把表达式理解成类似链表的结构：数字是节点，运算符连接节点。先遍历合并乘除节点，再合并加减节点。但这种结构化建模对本题过重。
思路 3：直接维护当前值计算。遇到加减时，需要判断后面是否有优先级更高的乘除，直接处理会比较麻烦，但可以做到 `O(1)` 额外空间。

最终在思路 3 的基础上收敛为分层解析：将连续乘除提取为 `getCountedNum`，外层只处理加减，从而自然消除所有最高优先级的乘除法。

## 卡点
- 实现前 30min 主要耗在方案收敛上：不是没有思路，而是思路太多且都能做，没有快速选中“无括号表达式用分层 parser”。
- debug 30min 的主要原因是成员变量和局部变量同名。最初为了避免到处传 `String s`，使用了成员变量 `private String s`，并写了 `this.s = s.replace(" ", "")`，但后续实际使用的仍然是参数 `s`，不是 `this.s`。误以为 `this.s` 会自然遮蔽局部变量，导致空格没有被真正去除，引发奇怪错误。
- 发现该问题后，又意识到 `replace` 会引入 `O(n)` 额外空间，于是改为在 `s2n` 中直接跳过空格。
- `while` 写法中，跳过空格分支和正常读数字分支都需要推进 `index`，容易出现重复推进或漏推进；最终改成 `for` 循环，让 `index++` 由循环统一负责。

## 关键点
- 本题最终采用分层解析：外层处理加减，内层 `getCountedNum` 处理连续乘除，`s2n` 处理数字。
- 对于无括号且只有 `+ - * /` 的表达式，应该优先收敛到 `expression -> term -> number` 的 parser 结构，不必先考虑双栈或构造中间结构。
- 如果成员变量和局部变量同名，直接写变量名会优先引用局部变量；成员变量必须显式使用 `this.xxx`。
- 为保持 `O(1)` 额外空间，不使用 `replace` 预处理空格，而是在解析数字时跳过空格。
- 当前偏好的 index 风格：尽量用 `for` 循环统一推进 `index`，减少不同分支内重复 `index++`。

## 复杂度
- 时间：`O(n)`
- 空间：`O(1)`

## 注意
- 最终解法没有问题，但耗时过长，主要暴露了方案收敛慢和 Java 变量遮蔽误判两个问题。
