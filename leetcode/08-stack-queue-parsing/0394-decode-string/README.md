# 394. Decode String

## 我的思考
一开始使用区间递归解析：`decode(s, begin, end)` 负责解析 `[begin, end)` 区间。遇到数字时，先解析重复次数，再通过 `findEnd` 找到匹配的右括号，递归解析括号内部字符串，最后重复拼接。

第一版可以通过，但 `findEnd` 每次都可能重新扫描括号区间，最坏情况下存在重复扫描。若输入长度为 `n`，输出长度为 `L`，第一版时间复杂度最坏为 `O(n^2 + L)`。

随后改为全局 `index` 递归解析。递归函数从当前 `index` 开始解析，遇到 `]` 停止并返回。这样每个输入字符基本只扫描一次，不再需要额外 `findEnd`，时间复杂度优化为 `O(n + L)`。

本题的关键不是算法方向，而是解析过程中 `index` 的推进责任。当前更偏好保留统一的循环末尾 `index++`，而不是在不同分支中重复推进。递归返回时，`index` 停在当前层的 `]` 上，由外层循环末尾的 `index++` 跳过该右括号。

## 卡点
- 一开始觉得循环内部需要手动推进 `index`，所以写成 `while`，但忘记 `index++`，导致死循环并最终栈溢出。
- 加上 `index++` 后，又陷入“分支内推进 index”与“统一循环末尾推进 index”的取舍。
- 最终选择保留统一推进。
- 第一版使用 `findEnd` 找匹配括号，虽然思路直观，但会带来重复扫描，复杂度更差。

## 关键点
- 第一版区间递归 + `findEnd`：最坏 `O(n^2 + L)`。
- 优化版全局 `index`：输入字符基本只扫描一次，复杂度为 `O(n + L)`。
- 当前写法的不变量：递归函数遇到 `]` 时停止，但不消费 `]`；调用层依靠循环末尾的 `index++` 跳过 `]`。
- 对本题而言，主要风险不是递归解析模型，而是 `index` 推进位置不清导致死循环、跳错字符或重复扫描。

## 复杂度
- 时间：优化版 `O(n + L)`，其中 `n` 为输入长度，`L` 为展开后的输出长度。
- 空间：`O(depth + L)`，其中 `depth` 为括号嵌套深度，`L` 为输出字符串长度。

## 注意
- 本题标记为 `x/r`，不是因为思路不会，而是因为暴露了字符串解析中 `index` 控制不够稳的问题。
