# v0.5 MVP 规划

## 1. 目标

v0.5 的目标是交付一个真实可用的日报总结闭环。系统每天固定时间拉取真实技术信息源内容，完成清洗、去重、筛选、排序、摘要和 Markdown 渲染，最终生成一份可直接阅读的日报。

该版本的核心价值是帮助使用者在 10 到 20 分钟内完成每日技术前沿浏览，并快速判断哪些内容值得深入阅读。

## 2. 交付标准

v0.5 达成时，应满足以下标准：

- 数据来自真实来源，而非 stub 或 mock 数据
- 每日流程可一次命令手动跑通
- 每次运行可抓取 30 到 100 条候选内容
- 去重与筛选后输出 5 到 8 条重点内容
- 每条重点内容包含标题、来源、链接、中文摘要和推荐理由
- 最终产出一份按日期命名的 Markdown 日报文件

## 3. 输出结构

日报建议采用以下结构：

- 今日概览
- Top 5 重点内容
- 值得深读
- 观察标签

其中：

- 今日概览说明抓取总量、去重后数量和最终入选数量
- Top 5 重点内容包含标题、来源、链接、2 到 4 句中文摘要、推荐原因
- 值得深读列出 1 到 3 条适合继续点击阅读的内容
- 观察标签用于标识当日主题分布，例如 `LLM`、`Agents`、`RAG`、`Infra`、`Reasoning`、`Open Source`

## 4. 范围

### 4.1 本版本包含

- 真实内容抓取
- 内容清洗与标准化
- 去重
- 相关性评分
- 重点内容筛选
- LLM 摘要生成
- Markdown 日报渲染
- 本地文件输出
- 手动触发执行

### 4.2 本版本不包含

- Web 前端
- 用户系统
- 多人订阅
- 数据库持久化
- 周报功能
- 复杂推荐解释系统
- 实时更新能力

上述内容不在 v0.5 范围内。本版本优先保证日报闭环可用，而非扩展外围能力。

## 5. 数据源规划

第一批数据源应控制在少量高价值来源，优先保证质量与稳定性，不追求覆盖面最大化。

建议首批接入以下来源：

- Hacker News
- arXiv 的 `cs.AI`、`cs.LG`、`cs.CL` RSS
- GitHub Trending 或高质量开源发布源
- 2 到 4 个技术博客 RSS
- OpenAI、Anthropic、Google DeepMind 等官方发布页或 RSS

数据源选择原则如下：

- 来源稳定
- 可通过 RSS 或结构化公开页面抓取
- 与技术前沿阅读场景直接相关
- 日均候选量可控，避免形成过高噪音

## 6. 设计边界

系统采用确定性逻辑优先、LLM 辅助的设计。

LLM 负责：

- 摘要生成
- 标签补充
- 可选的推荐理由生成

非 LLM 负责：

- 抓取
- 解析
- 标准化
- 去重
- 过滤
- 排序
- 失败降级

该边界用于保证结果稳定性、可解释性和运行成本可控。系统不应将内容选择责任完全交由 LLM 决定。

## 7. 模块落地计划

### 7.1 SourceFetcher 实现

先实现基于 RSS 的真实抓取能力。

需要完成：

- 增加可配置的数据源清单
- 实现 `RssFetcher`
- 将抓取结果统一映射为 `RawItem`

第一阶段不要求覆盖所有来源，但必须保证至少一批真实源可稳定抓取。

### 7.2 NormalizationService 实现

需要完成：

- 标题、URL、发布时间、正文内容的清洗
- 不同来源的结构统一
- 统一输出为 `NormalizedItem`
- 提供初版标签提取能力

初版标签提取可采用关键词规则，不要求一开始依赖 LLM。

### 7.3 DeduplicationService 实现

需要完成：

- 基于 URL 的直接去重
- 基于标题近似匹配的重复内容合并

目标是避免同一事件被多个源重复收录，降低日报冗余度。

### 7.4 RelevanceScoringService 实现

需要完成：

- 来源权重评分
- 标题关键词评分
- 主题相关性评分
- 发布时间新鲜度评分

评分逻辑应先采用确定性规则实现，以便调试和快速修正。

### 7.5 SummarizationService 实现

需要完成：

- 对入选前 N 条内容调用 Gemini CLI
- 输出简洁中文摘要
- 生成必要的推荐理由

该模块必须提供失败降级能力。若 LLM 调用失败，系统仍应保留标题、来源、链接和原始简述，不应导致整份日报失败。

### 7.6 DailyDigestGenerator 与 MarkdownRenderer 实现

需要完成：

- 将筛选结果转换为日报结构
- 输出可直接阅读的 Markdown
- 产物落盘到本地输出目录

输出文件应按日期命名，便于历史保存和回看。

### 7.7 DailyJob 实现

需要完成：

- 串联抓取、处理、摘要、渲染和输出链路
- 提供可手动触发的运行入口

该入口是 v0.5 的主执行路径。定时调度后续补充。

## 8. 实施阶段

### 阶段 1：真实抓取闭环

目标：

- 建立 RSS 数据源配置
- 完成真实抓取
- 输出原始候选内容

阶段交付物：

- 手动执行后能够得到一份真实抓取结果

### 阶段 2：筛选闭环

目标：

- 完成标准化
- 完成去重
- 完成相关性评分
- 选出候选阅读列表

阶段交付物：

- 一份经过筛选的重点候选列表

该阶段暂不依赖 LLM，优先验证筛选质量是否足够支撑日报场景。

### 阶段 3：摘要闭环

目标：

- 接入 Gemini CLI
- 对入选内容生成中文摘要
- 输出完整 Markdown 日报

阶段交付物：

- 第一份真实可读的日报文件

### 阶段 4：自动化闭环

目标：

- 增加定时执行
- 增加运行日志
- 增加失败处理与重试策略
- 增加日报历史保存约定

阶段交付物：

- 每日自动生成的稳定日报流程

## 9. 风险与约束

v0.5 的主要风险如下：

- 数据源质量不足会直接降低日报价值
- 去重不准会导致同类内容反复出现
- 筛选不准会使日报退化为链接堆砌
- 抓取到的正文质量不足时，摘要效果会明显下降
- Gemini CLI 调用不稳定时，必须保证系统能够降级输出
- 日报条目过多会降低每日阅读效率

因此，v0.5 的重点不是“抓更多”，而是“抓准、筛准、总结准”。

## 10. 验证计划

v0.5 的实现验证应覆盖以下方面：

- 抓取流程能够稳定获取真实源内容
- 去重后条目数量明显下降且不误删主要内容
- 排序结果能稳定选出高价值主题
- 摘要结果语言可读、内容不空泛
- Markdown 产物结构完整，可直接阅读
- 单次运行在可接受时间内完成

## 11. 后续递延项

以下内容明确递延，不属于 v0.5：

- 周报生成
- 多源更复杂解析器
- 数据库存储
- Web 展示界面
- 个性化偏好配置
- 多人使用场景

这些内容应在日报主链路稳定后再决定是否进入后续版本。
